# 🎮 게임 요구사항 정의서 (PRD) - v1.12

## 1. 게임 개요 (Overview)

* **장르**: 2D 탑뷰 AI 텍스트 컨트롤 어드벤처
* **그래픽 스타일**: 2D 픽셀 아트 (추후 AI 이미지 생성기를 통해 에셋 제작 예정)
* **플랫폼**: 웹 브라우저 (Web Game)
* **핵심 컨셉**: 플레이어는 캐릭터를 직접 조작할 수 없으며, **오직 텍스트 채팅**을 통해서만 지시를 내립니다. 캐릭터는 자연어 처리(GPT)를 통해 플레이어의 말을 이해하고 대답하며, 그에 맞는 행동을 수행합니다.
* **최종 목표**: 시작 맵에서 자원을 채집해 검을 제작/강화하고, 최종 맵으로 이동하여 마왕을 토벌하는 것.

## 2. 핵심 게임 루프 (Core Game Loop)

1. **지시**: 플레이어가 채팅창에 자연어로 명령 입력 (예: "저기 보이는 나무 쪽으로 가서 나무 좀 베어줄래?")
2. **AI 처리**: OpenAI API를 통해 두 가지 결과를 도출
   * **대사**: 캐릭터의 성격이 반영된 자연스러운 텍스트 대답 (예: "알겠어! 나무를 베러 갈게. 조금만 기다려.")
   * **명령어**: 사전 정의된 시스템 커맨드로 변환 (예: `MOVE_OBJ(TREE)`, `INTERACT(TREE)`)
3. **실행**: 서버 검증을 거친 커맨드를 클라이언트로 전달하여 애니메이션과 함께 실행 (이동 및 상호작용).
4. **성장**: 획득한 자원(나무, 철)으로 모루에서 검을 1차 제작.
5. **중간 보스**: 드래곤을 처치하여 '드래곤의 정수'를 획득하고 검을 최종(2차) 강화.
6. **도전**: 최종 맵으로 이동해 마왕과 결전.
7. **결과**: 실패 시 **게임 오버 및 처음부터 재시작**, 성공 시 엔딩.

## 3. 시스템 및 UI 요구사항 (System & UI Requirements)

### 3.1. 채팅 및 명령 시스템

* **통신 방식**: 웹소켓(WebSocket)이 아닌 **HTTP 요청/응답(Request-Response) 구조** 사용.
* **입력 제어**: 플레이어가 명령을 입력하고, 캐릭터가 해당 명령의 동작(이동, 상호작용 등)을 **완전히 수행하기 전까지 채팅창 입력이 잠금(Block) 처리**됨.
* **말풍선 UI (Speech Bubble)**:
  * 캐릭터의 모든 대답은 화면 하단 채팅창뿐만 아니라 **캐릭터 머리 위의 말풍선** 형태로도 즉시 출력됩니다.
  * 서버(GPT API) 통신 중 발생하는 딜레이 시간 동안에는, 플레이어가 지루하지 않고 처리 중임을 인지할 수 있도록 캐릭터 머리 위에 **"생각 중..."** 이라는 말풍선(또는 로딩 애니메이션)을 띄워줍니다.

### 3.2. 아트 스타일 및 UI 디자인

* **디자인 컨셉**: 마인크래프트(Minecraft)와 유사한 직관적이고 레트로한 픽셀 디자인.
* **1프레임 기반 애니메이션**:
  * 캐릭터(플레이어블 포함) 및 몬스터, 오브젝트에 대해 별도의 애니메이션 시트(Sprite Sheet)를 사용하지 않습니다.
  * **모든 에셋은 1프레임의 단일 이미지로 제공**됩니다.
  * 생동감을 주기 위해 클라이언트 코드(JS/CSS)를 활용하며, 특히 **캐릭터가 이동할 때 이미지가 위아래로 살짝 들썩이는 효과(Bobbing)**를 주어 자연스럽게 걷는 듯한 애니메이션을 표현합니다.
* **인벤토리 UI**:
  * 화면 하단 또는 우측에 픽셀 테두리를 가진 슬롯(Grid) 형태로 상시 노출.
  * 자원(나무, 철, 드래곤의 정수) 획득 시 슬롯에 픽셀 아이콘과 누적 개수가 표시됨.
  * 현재 장착 중인 검의 상태(맨손 -> 1차 검 -> 2차 검)를 보여주는 전용 장비 슬롯 제공.
* **시스템 UI (재시작 버튼)**:
  * 화면 상단 또는 모서리에 **'재시작(Restart)' 버튼 상시 노출**.
  * 플레이어가 필수 오브젝트를 파괴하여 게임 진행이 불가능한 '소프트락(Soft-lock)' 상태에 빠졌을 때 언제든 맵을 초기화할 수 있도록 지원.

### 3.3. 이동 시스템

* **이동 방식 1 (방향/거리)**: 지정된 방향으로 N칸 이동. (예: `MOVE_DIR(UP, 3)`)
* **이동 방식 2 (오브젝트 지정)**: 특정 오브젝트까지 최단 거리로 자동 이동. (예: `MOVE_OBJ(ANVIL)`)

## 4. 콘텐츠 및 오브젝트 (Content & Objects)

*주의: 바닥, 벽과 같은 1차 레이어(배경)는 상호작용 및 공격 대상이 되는 '오브젝트'로 취급하지 않습니다.*

### 4.1. 맵 (Maps)

* **Map 1 (시작 맵)**: 평화로운 구역. 나무, 광맥, 모루, 드래곤이 존재. 자원 채집과 성장이 이루어지는 공간.
* **Map 2 (마왕의 성)**: 플레이어가 원할 때 이동 가능. 마왕이 존재하는 보스전 전용 공간.

### 4.2. 리소스 및 구조물 (상호작용/공격 규칙)

모든 오브젝트는 `INTERACT`와 `ATTACK` 두 가지 커맨드에 대해 다르게 반응합니다.
*파괴(`ATTACK`) 시 오브젝트가 맵에서 완전히 소멸하지 않고, 파괴된 형태의 에셋으로 변경되어 맵에 남습니다.*

* **나무 (Tree)**
  * `INTERACT`: 나무 채집 (인벤토리에 '나무' 추가).
  * `ATTACK`: 나무 파괴 (자원 획득 불가, '파괴된 그루터기' 에셋으로 변경).
* **광맥 (Ore Vein)**
  * `INTERACT`: 광석 채집 (인벤토리에 '철' 추가).
  * `ATTACK`: 광맥 파괴 (자원 획득 불가, '고갈된 광맥' 에셋으로 변경).
* **모루 (Anvil)**
  * `INTERACT`: 현재 보유 재료 및 검의 상태를 체크하여 1차 제작 또는 2차 강화를 시도합니다. (재료 부족 시 UI 경고음 및 대사로 안내)
    * **1차 (검 제작)**: 나무 3개 + 철 3개 소모.
    * **2차 (최종 강화)**: 드래곤의 정수 1개 소모.
  * `ATTACK`: 모루 파괴. 파괴 시 더 이상 검을 제작하거나 강화할 수 없어 **게임 클리어가 불가능한 소프트락(Soft-lock) 상태**가 됩니다. 유저는 스스로 UI의 '재시작' 버튼을 눌러야 합니다.

### 4.3. 몬스터 전투 및 판정 규칙

전투는 별도의 턴제 전투 없이 **공격(`ATTACK`) 시 현재 검의 강화 상태에 따라 승패가 즉시 판정**됩니다. 패배 시 즉시 **게임 오버 처리되며 Map 1 초기 상태에서 재시작**합니다.

* **드래곤 (Map 1 중간 보스)**
  * `INTERACT`: 전투를 진행하지 않고 드래곤의 고유 대사 출력.
  * `ATTACK`: 전투 판정 진행. (승리 조건: '1차 검' 장착 상태)
* **마왕 (Map 2 최종 보스)**
  * `INTERACT`: 전투를 진행하지 않고 마왕의 고유 대사 출력.
  * `ATTACK`: 전투 판정 진행. (승리 조건: '2차 검' 장착 상태)

## 5. 명령어 사전 초안 (Command Dictionary)

*모든 채집, 제작, 대화 기능이 `INTERACT`로 통합되었습니다.*

* `MOVE_DIR(direction, distance)`: 방향(UP, DOWN, LEFT, RIGHT)으로 N칸 이동
* `MOVE_OBJ(object_id)`: 특정 오브젝트로 최단거리 이동
* `INTERACT(object_id)`: 특정 오브젝트와 상호작용 (채집, 제작/강화 시도, NPC 대사 출력 등)
* `ATTACK(object_id)`: 특정 오브젝트 공격 (몬스터는 전투 판정, 리소스/구조물은 파괴)
* `MOVE_MAP(map_id)`: 다른 맵으로 이동
* `IDLE()`: 지시가 불명확하거나 실행할 수 없을 때 대기

## 7. 기술 스펙 및 아키텍처 (Technical Specifications)

### 7.1. 인프라 및 환경 (Infrastructure)

* **배포 및 컨테이너화**: 배포 관리의 용이성과 환경 일관성을 위해 **Docker-compose**를 사용하여 구축 및 실행.
* **네트워크 환경**: 로컬 테스트 및 운영 환경을 모두 지원하기 위해 **HTTP와 HTTPS 통신이 모두 동작**하도록 설정.

### 7.2. 서버 아키텍처 (Backend)

* **프레임워크**: **Node.js (Express.js)**
* **모듈화 전략**: 역할에 따라 파일을 명확히 분리하여 유지보수성 극대화.
  * 엔드포인트(라우터) 관리 파일
  * GPT API 호출 로직 담당 파일
  * 문자열 파싱 및 커맨드 정제 로직 담당 파일 등

### 7.3. 클라이언트 아키텍처 (Frontend)

* **에셋 로딩**: 게임 시작 시 모든 정적 자원(단일 프레임 이미지, 사운드 등)을 메모리에 미리 로드(Pre-load)하여 플레이 중 지연 방지.
* **스크립트 모듈화**: 역할 기반으로 JavaScript 파일을 분리하여 작성.
  1. **Network (통신)**: 서버와의 HTTP 요청/응답 처리 로직
  2. **Command - Move (이동)**: 캐릭터의 타일/좌표 계산 진행. 목적지 이동 시 CSS/JS 기반의 트윈(Tween)을 적용하며, **사인파(Sine wave)를 이용해 상하로 살짝 들썩이는 모션(Bobbing)**을 주어 걷는 애니메이션을 연출합니다.
  3. **Command - Interact (상호작용)**: 아이템 획득, 모루 조작, 대사 출력 및 소소한 이미지 튕김 이펙트 처리.
  4. **Command - Attack (공격)**: 대상 공격 시 '플레이어 공격 포즈(CHR_02)' 이미지로 일시 교체되며 급가속 돌진, 셰이크(Shake) 등 모션 적용.
  5. **State Management (상태 관리)**: 캐릭터의 현재 위치, 방향(좌우 반전 상태), 인벤토리 현황, 장착된 검 상태 등 데이터 관리.

### 7.4. LLM 파이프라인 프로세스

* **모델**: `gpt-5.4-nano` 사용 (빠른 추론과 비용 효율성 도모)
* **프롬프트 관리**: 줄바꿈 및 복잡한 구조화가 필요한 모든 시스템 프롬프트 및 지시문은 `.env`가 아닌 **별도의 `prompt.json` 파일로 관리**합니다. 프롬프트 수정 사항을 서버 재시작 없이 즉각 반영하기 위해, **서버는 매 API 요청이 발생할 때마다 `prompt.json` 파일을 새로 읽어와(Read) 프롬프트를 주입**합니다.
* **처리 파이프라인 (2-Step API Call)**:
  1. **1차 GPT 호출**: 사용자의 입력 채팅을 분석하여 사전에 정의된 '동작 커맨드' 문자열을 추출.
  2. **커맨드 정제 (Server Logic)**: 반환된 문자열에서 불필요한 문자를 제거하고 정규식 등을 통해 정확한 규격의 커맨드 형태로 파싱 및 검증.
  3. **2차 GPT 호출**: 정제(확정)된 커맨드와 플레이어의 원본 채팅을 다시 GPT에 주입하여, 해당 상황과 지시에 알맞은 '캐릭터의 자연스러운 대답' 생성.
  4. **응답 반환**: 정제된 커맨드 배열과 캐릭터의 대답을 JSON 형태로 묶어 클라이언트로 응답(Response).

## 8. 그래픽 에셋 리스트 (Graphic Asset List)

1프레임 애니메이션 및 UI에 사용될 2D 픽셀 아트 에셋 리스트입니다. 에셋이 존재하지 않는 디자인은 HTML + CSS 로 구현합니다.

> **💡 실제 이미지 파일 크기 참고사항**
> 아래 리스트에 기재된 크기(예: 64x64, 32x32)는 게임 내에서 차지하는 '픽셀 논리적 크기(권장 렌더링 사이즈)'를 의미합니다. AI로 생성된 실제 원본 파일의 해상도는 다음과 같이 훨씬 크며, 클라이언트에서 CSS/Canvas 스케일링을 통해 사이즈를 맞춰 사용합니다.
>
> * **기본 에셋 (캐릭터, 몬스터, 오브젝트, 아이콘, UI_01)**: `500 x 500 px`
> * **타일셋 (TLS_01, TLS_02)**: `1024 x 1024 px`
> * **재시작 버튼 (UI_02)**: `711 x 351 px`

### 8.1. 캐릭터 및 몬스터 (Characters & Monsters)

* **`CHR_01` (플레이어 캐릭터)**: 정지화 (64x64) - 기본 대기 상태
* **`CHR_02` (플레이어 공격 포즈)**: 단일 모션 (64x64) - 검을 휘두르는 모션 및 검기 이펙트 포함 프레임
* **`MON_01` (드래곤)**: 정지화 (128x128) - Map 1 중간 보스
* **`MON_02` (마왕)**: 정지화 (128x128) - Map 2 최종 보스

### 8.2. 맵 오브젝트 (Map Objects)

* **`OBJ_01` (나무)**: (64x64) - 채집 가능한 온전한 나무
* **`OBJ_02` (파괴된 나무 그루터기)**: (64x64) - ATTACK 커맨드로 파괴된 후의 상태
* **`OBJ_03` (광맥)**: (64x64) - 채집 가능한 온전한 광맥
* **`OBJ_04` (고갈된 광맥)**: (64x64) - 파괴(ATTACK) 또는 채집 후 빈 바위 상태
* **`OBJ_05` (모루)**: (64x64) - 무기 제작 및 강화를 위한 제작대

### 8.3. 아이템 및 인벤토리 아이콘 (Item Icons)

* **`ITC_01` (나무 리소스)**: (32x32) - 잘린 장작 묶음 이미지
* **`ITC_02` (철 리소스)**: (32x32) - 빛나는 회색 철광석 이미지
* **`ITC_03` (드래곤의 정수)**: (32x32) - 붉은색 마력의 구슬 형태 이미지
* **`ITC_04` (맨손 상태)**: (32x32) - 주먹 쥔 손 이미지 (초기 무기 슬롯)
* **`ITC_05` (1차 검)**: (32x32) - 철제 숏소드 이미지

## 9. 맵 디자인 및 배치 정보 (Map Design & Layout)

PC 웹 브라우저 환경에서 스크롤 없이 한눈에 맵 전체를 파악할 수 있도록, 맵은 **정사각형 형태의 콤팩트한 규모(예: 15 x 15 그리드 타일)**로 구성합니다. 플레이어가 이동(보빙 애니메이션) 결과를 기다리는 시간을 최소화하고 쾌적하게 플레이할 수 있는 크기입니다.

### 9.1. Map 1 (시작 맵 - 평화의 숲)
* **크기**: 15 x 15 타일 (논리적 크기 64x64 기준, 총 960x960 해상도 차지)
* **배치 목적**: 자원 채집과 1차 무기 제작, 중간 보스 토벌이 이루어지는 메인 공간
* **배치 구조**:
    * **플레이어 시작점**: 맵의 최하단 중앙에 위치.
    * **제작대 (모루 `OBJ_05`)**: 맵의 정중앙 또는 시작점 바로 앞 안전 구역에 배치하여 접근성을 높임.
    * **자원 (나무 `OBJ_01` / 광맥 `OBJ_03`)**: 맵의 좌우 측면 및 모서리 주변에 불규칙하게 산개하여 배치.
    * **중간 보스 (드래곤 `MON_01`)**: 맵의 최상단 중앙에 배치. 드래곤의 바로 뒤편에는 마왕의 성(Map 2)으로 진입할 수 있는 입구(포탈)가 존재함.

### 9.2. Map 2 (최종 보스 맵 - 마왕의 알현실)
* **크기**: 15 x 15 타일 (또는 몰입감을 위해 11 x 11로 약간 축소된 투기장 형태)
* **배치 목적**: 오직 마왕과의 최종 전투(스탯 판정)만을 위한 단독 공간
* **배치 구조**:
    * **마왕 (`MON_02`)을 제외한 어떠한 자원이나 모루도 배치하지 않음.**
    * **플레이어 진입점**: 맵의 최하단 중앙.
    * **최종 보스 (마왕 `MON_02`)**: 맵의 최상단 중앙에 압도적인 존재감으로 배치되며, 플레이어와 마왕 사이의 중앙 공간은 이동에 방해되는 장애물 없이 완전히 비워둠.
